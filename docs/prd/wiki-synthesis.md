# PRD: Wiki Page Synthesis

## Problem Statement

The ingestion pipeline currently parses sources, chunks them, embeds them, and extracts entities and relations — but stops before creating or updating wiki pages. The `WikiPage` table exists in Postgres (per ADR-0006) but is entirely unused. The system has no mechanism to turn the knowledge graph into LLM-maintained markdown pages. Users cannot browse synthesized knowledge, and the gap between graph data and human-readable output makes the system incomplete for its stated purpose.

## Solution

Add a **wiki page synthesis pipeline** that runs after source ingestion completes. For every entity touched by a source, it creates or updates a `WikiPage` row using an LLM that synthesizes prose from source chunks, preserves existing content, and records contradictions. Each source also gets a source-summary page. Synthesis is asynchronous, decoupled from ingestion via the existing Postgres job queue (ADR-0005), with claim-time job coalescing so that multiple sources touching the same entity result in a single synthesis call.

## User Stories

1. As a system operator, I want entity wiki pages to be automatically created when a source is ingested, so that the knowledge graph becomes human-readable.
2. As a system operator, I want entity pages to be updated (not overwritten) when new sources mention the same entity, so that knowledge accumulates rather than gets replaced.
3. As a system operator, I want contradictions between sources to be surfaced in the wiki page, so that conflicting claims are visible and traceable.
4. As a system operator, I want citations on every factual claim in the wiki, so that I can verify information against the original source chunks.
5. As a system operator, I want a source-summary page for every ingested document, so that I can see at a glance what each source contributed.
6. As a system operator, I want the synthesis pipeline to handle entity page updates safely even when multiple sources are ingested concurrently, so that no content is lost.
7. As a system operator, I want synthesis failures to be isolated (one failing page doesn't break the entire source), so that partial failures are recoverable.
8. As a system operator, I want the synthesis jobs to be coalesced when multiple sources touch the same entity, so that the same entity isn't synthesized multiple times redundantly.
9. As a developer, I want the synthesis worker flow to use the same advisory locking pattern as entity resolution, so that concurrency behavior is consistent.
10. As a developer, I want the synthesis prompt to be a single Jinja2 template with conditional blocks for create vs update, so that prompt maintenance is minimal.
11. As a developer, I want the context assembly to use a tiered token budget with hard slots, so that the LLM always gets the most important context first.
12. As a developer, I want relations to be rendered programmatically from the database rather than generated by the LLM, so that the relations section is always accurate.
13. As a developer, I want backlink join table population to be graph-based only, so that the mechanism is deterministic and testable.
14. As a system operator, I want the wiki pages to have stable, human-readable slugs, so that they can be browsed and linked externally.
15. As a developer, I want synthesis errors to be logged with full context, so that debugging a failed synthesis is straightforward.

## Implementation Decisions

### 1. Architecture: separate synthesis jobs with coalescing

After a source completes ingestion, the pipeline enqueues one `synthesize_wiki_page` job per entity that the source touched, plus one `synthesize_source_summary` job for the source itself. These are separate from the ingest job so that ingestion is not blocked by LLM synthesis time.

A separate synthesis worker claims these jobs. When claiming a `synthesize_wiki_page` job, the worker looks for other pending jobs with the same `target_entity_id` (and same `job_type`), merges their `source_ids` into the claimed job's payload, marks the duplicates as `cancelled`, and then proceeds to synthesize once with the merged source list.

`source_summary` jobs are never coalesced — each source gets exactly one summary page.

### 2. Schema changes

**`wiki_pages` table** (two new columns):
- `synthesized_from_sources`: JSONB array of source UUID strings — audit trail of which sources contributed to this page's current version.
- `synthesized_at`: nullable timestamp — when the current content was last synthesized.

**`jobs` table** (one new column):
- `target_entity_id`: nullable UUID, indexed — the entity that a `synthesize_wiki_page` job targets. Enables the coalescing query.

### 3. Concurrency model: advisory lock per entity

The synthesis worker, after claiming and merging a job, acquires a Postgres advisory lock on a hash of the entity's UUID. It retries `pg_try_advisory_lock` with exponential backoff. If retries are exhausted, it releases the job claim (marking it back to `pending`) so another worker can pick it up later.

Inside the advisory lock, the worker re-reads the current `WikiPage` (Slot 3 context), builds the full context, calls the LLM, writes the page, releases the lock, and marks the job `completed`.

### 4. Update strategy: merge-with-diff

The LLM prompt instructs: preserve all existing content unless directly contradicted by a new source chunk. Where contradicted, update the text and record the contradiction. Never remove existing detail to shorten the page. This is a single prompt with a conditional block that changes wording based on whether `existing_page` is present.

### 5. Context assembly: tiered token budget (6,000 tokens)

| Tier | Slot | Budget | Content |
|------|------|--------|---------|
| 1 | Synthesis instructions + entity metadata | ~500 tok | Prompt preamble + entity name, type, description |
| 2 | Existing wiki page (if updating) | ≤800 tok | Head-truncated current content |
| 3 | All relations (programmatic) | ~200 tok | Rendered from `relations` table |
| 4 | Source chunks | remaining | Scored by embedding similarity, cosine-deduplicated, truncated to budget |
| 5 | Known entities / stubs | if room | 1-hop neighbors + string-match pre-scan |

The source chunks are ranked by embedding similarity to the entity description, deduplicated by cosine similarity, and filled until the 6k budget is exhausted.

### 6. Page content structure

**Entity page:**
- `#{title}` heading with entity name
- `**Type:** {type}` badge
- 1–3 paragraphs of prose synthesis, each claim cited `[file_name:chunk_index]`
- `## Relations` — bullet list of `relation_type → [[target-entity-slug]]`, rendered programmatically from DB
- `## Contradictions` — explicit conflict notes, or `*No contradictions detected.*`
- `## Sources` — list of source files with contributing chunk counts

**Source-summary page:**
- `#{file_name}` heading
- `**Source type:** {type}  **Ingested:** {date}`
- 2–4 paragraphs of prose synthesis
- `## Pages created or updated` — links to touched entity pages
- `## Key relations introduced` — flat list (no confidence distinction in v1)
- `## Contradictions` — explicit conflicts
- `## Ingest history` — metadata
- `## Chunks` — full listing

### 7. Slug generation

`slug = f"{slugify(name)}-{entity_id[:8]}"` — human-readable, deterministic, collision-free.

### 8. Backlink population

After writing `WikiPage.content`, populate `wiki_page_entities` join table by inserting all entities connected by `incoming_relations` or `outgoing_relations` to this page's entity. Graph-based only — no parsing of LLM output for wiki links.

### 9. Error handling

- Entity page synthesis: retry once on transient LLM error, then skip (log and continue). The entity's data is already in the graph; missing wiki page can be regenerated later.
- Source-summary synthesis: same retry-once-then-skip.
- Enqueue failure during ingestion: log and continue; the source is still marked `PROCESSED`. Missing synthesis jobs can be backfilled via CLI or lint pass.

### 10. Integration point in ingestion pipeline

In `pipeline.py`, after the chunk-processing loop completes (and before the source-status block), collect all touched entity IDs from successfully processed chunks, then enqueue one `synthesize_wiki_page` job per entity and one `synthesize_source_summary` job for the source.

### 11. Worker flow (pseudocode)

1. Claim job (`SELECT ... FOR UPDATE SKIP LOCKED` → status `claimed`)
2. Query for pending duplicates (`same job_type + same target_entity_id`)
3. Merge their `source_ids` into claimed payload; mark duplicates `cancelled`
4. COMMIT (steps 1–3 as one transaction)
5. Acquire advisory lock on entity_id hash (retry loop with backoff)
6. If lock fails after retries: release claim → `pending`; return
7. Re-read current `WikiPage` inside advisory lock
8. Fetch chunks scoped to merged source_ids
9. Build context (structural anchor, 1-hop subgraph, existing page, chunks)
10. LLM synthesize
11. Write `WikiPage` (content, synthesized_from_sources, synthesized_at)
12. Release advisory lock
13. Mark job `completed` (separate transaction)

### 12. Prompt templates

Two Jinja2 templates:
- `synthesize_entity.j2` — single prompt, conditional on update vs create, with 8 explicit rules (chunks only, no citation fabrication, no confidence laundering, no silent contradiction resolution, no stub inflation, wiki links closed set, preserve existing content, no filler).
- `synthesize_source_summary.j2` — separate short prompt, no entity concept, 7 rules.

Both stripped of confidence-tagging (EXTRACTED/INFERRED) and graph centrality metrics for v1.

## Testing Decisions

### What makes a good test

Tests should verify **external behavior** — what the synthesis worker does given specific inputs — not **implementation details** such as the exact prompt string or the number of DB queries. We test that a claimed job produces a written `WikiPage` row with expected content, or that a duplicate job gets cancelled, not that the prompt template was rendered with `{{ entity_name }}`.

### Modules to test

1. **Synthesis worker** (`tests/wiki/test_synthesis.py`):
   - Claim job → merge duplicate → synthesize → write page → complete job
   - Advisory lock contention → retry loop → eventual success
   - Advisory lock exhaustion → release claim back to pending
   - LLM error → retry once → skip on second failure
   - Source-summary synthesis flow

2. **Context assembly** (`tests/wiki/test_context.py`):
   - Tiered budget enforcement (6k limit)
   - Chunk scoring by embedding similarity
   - Cosine deduplication of chunks
   - Known entities list bounded to neighbors + string-match

3. **Prompt rendering** (`tests/wiki/test_prompts.py`):
   - Render create prompt (no existing page)
   - Render update prompt (with existing page)
   - Render source-summary prompt
   - Verify all rules appear in output

4. **Ingest integration** (`tests/ingest/test_pipeline.py` modified):
   - After pipeline run, assert correct synthesis jobs enqueued
   - Assert source-summary job enqueued
   - Assert enqueue failure is logged, not raised

### Prior art

- `tests/graph/test_extraction.py` — uses `FakeChatProvider` to mock LLM responses; same pattern for synthesis tests.
- `tests/graph/test_resolution.py` — tests advisory lock pattern and graceful degradation; same pattern for synthesis worker tests.
- `tests/jobs/` — tests job queue mechanics (claim, complete, fail); synthesis worker tests extend this pattern.

## Out of Scope

- **Confidence tagging on relations** — EXTRACTED/INFERRED distinction and confidence scores on `relations` rows. Deferred to v2.
- **Graph centrality metrics** — centrality rank, community clustering, PageRank, Leiden. Deferred to v2.
- **Coverage gaps / stub index** — automatic detection and listing of entities mentioned but lacking wiki pages. Deferred to v2.
- **Post-synthesis merge review** — automatic lint pass on entities synthesized more than N times to flag contradictory sources for human review. Flagged for future work.
- **File export to Obsidian** — optional `.md` export per ADR-0006. Not part of synthesis; can be added later as a separate job type.
- **Wiki link parsing from LLM output** — we only render relations programmatically from the graph.
- **Manual re-synthesis CLI** — `rag-wiki re-synthesize` command. Useful for backfill but not required for the core pipeline.
- **Source invalidation / deletion** — handling corrected or removed sources and triggering re-synthesis. Needs design but not part of this PRD.

## Further Notes

- The `LLM_MODEL_WIKI_SYNTHESIS` env var is already configured in `settings.py` but unused. No new env vars needed for v1.
- The `jobs` table already has `payload` (JSONB) and `status` columns. The new `target_entity_id` column is the only schema change for the job queue.
- The `wiki_pages` table already has `entity_id` (nullable), `title`, `slug`, `content`, and `status`. The two new audit columns are additive.
- Existing code in `pipeline.py` already handles chunk-level graceful degradation; the synthesis enqueue hook follows the same philosophy.
- The synthesis worker reuses the same `ChatProvider` protocol as extraction and resolution; no new provider interface needed.
